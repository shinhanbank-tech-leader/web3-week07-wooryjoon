## 스마트컨트랙트 업그레이드 방식과 은행 시스템 변경 방식 비교 분석

## 레포트 내용

### "수정할 수 없는 코드" vs "끊임없이 변하는 비즈니스"

## 스마트컨트랙트의 업그레이드 방식
스마트컨트랙트는 기본적으로 한번 배포되면 수정할 수 없는 불변성을 띈다. 이 불변성은 신뢰의 근간이 되지만, 버그 수정이나 기능 개선이 필요한 운영 관점에서는 치명적인 약점이 될 수 있다.
컨트랙트를 고칠 수 있는 유일한 방법은, 블록체인 내의 새로운 주소에 컨트랙트를 배포하는 것 뿐이다. 하지만 이 경우에는 기존의 컨트랙트에 담긴 데이터를 유지할 수 없게 되는 문제가 생긴다.


## 은행 시스템의 변경 방식
은행 시스템은 형상관리라고 불리는 코드 관리 시스템에 의해 유지되고 운영된다. 코어뱅킹, 전자금융 등의 모든 전문과 화면 코드는 형상관리에 의해 변화된다.
변경사항이 발생할 경우, 변경사항에 대해서 품질검증팀의 검증을 받아야하며, 연관된 모든 부서와 팀의 영향분석 테스트를 통과해야 배포할 수 있다.
스마트컨트랙트의 업그레이드 방식보다는 유연하다고 볼 수 있고, 인간의 통제 및 승인 하에 이루어지는 프로세스이다.


## 스마트컨트랙트를 은행 시스템에 적용한다면 발생할 문제점
스마트컨트랙트를 은행 시스템에 적용한다면 겪게 될 가장 큰 문제점은, '유연함의 부족' 이다.
정부 및 각종 금융 기관의 비즈니스적 요구를 끊임없이 발생하기 때문에 수시로 로직을 변경해야 하며 하루에도 수십개의 체크인과 이행이 필요한 반면, 스마트컨트랙트는 이러한 잦은 변경을 감당하기에 유연한 구조라고 볼 수 없다.

## 스마트컨트랙트의 해결책 : Proxy와 delegateCall
이 문제를 해결하기 위해 시스템을 데이터 저장소(Proxy)'와 '실행 로직(Implementation)'으로 분리한다.

Proxy 계약 (State): 모든 데이터(Variable)를 저장하고 사용자와 통신하는 고정된 주소.

Implementation 계약 (Logic): 실제 동작하는 코드가 담긴 계약.

delegateCall: Proxy가 사용자의 요청을 받으면 delegateCall을 통해 로직 계약의 코드를 불러와 실행한다. 이때 코드는 로직 계약의 것을 쓰지만, 데이터는 Proxy의 저장 공간에 기록된다.

은행 운영 관점의 의미: 이는 마치 은행의 DB를 그대로 두고, 그 데이터를 처리하는 서버 로직만 교체하는 것과 흡사하다. 사용자는 항상 같은 API(주소)를 호출하지만, 내부 로직은 유연하게 변경되는 구조이다.

## proxy 방식의 한계
가장 큰 한계는 롤백이다. 업그레이드 과정에서 문제가 발생한 경우, 롤백은 필수적이다. 은행의 기존 시스템에서는 롤백을 통해 서버와 DB를 이전 스냅샷으로 되돌림으로서 리스크를 최소화할 수 있지만,

스마트컨트랙트가 잘못된 경우, 로직을 롤백할 수는 있겠지만, 그 기간 동안 수행된 트랜잭션 자체를 되돌리는 것은 굉장히 힘들 것이다. 금융 시스템은 실제로 돈이 움직이는 환경이기 때문에,
한번 한번의 실수가 치명적인 리스크를 초래하게 된다. 

## 결론
끊임없이 발생할 비즈니스적 요구사항에 대해 현재의 은행 시스템이 아닌 스마트컨트랙트 방식으로 유연하게 대응할 수 있을지는 잘 모르겠다. proxy 방식 등 여러 방법을 통해 현실세계의 규제에 맞는 유연함을 갖출 수 있는 형식들을 많이 고민해봐야할 것 같다.
